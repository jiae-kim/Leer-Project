### 📚개요

 - 팀 소개 : 잡지읽어조 (6인)
   
   - 팀장 : 노희영
   
   - 팀원 : 김지애, 김은지, 김범진, 박상현, 정다혜

 - 프로젝트명 : Leer
 
 - 개발 일정 : 2022.06.27 ~ 2022.08.08
 
   - 기획 및 요구사항 분석 : 2022.06.27 ~ 07.04
   
   - UI 설계 : 2022.07.05 ~ 07.12
   
   - DB 설계 : 2022.07.13 ~ 07.19
   
   - 프로젝트 구현 및 디버깅 : 2022.07.20 ~ 08.05
   
   - 프로젝트 발표 : 2022.08.08
 
 - 기획 의도 
 
   - 잡지 구매 시 대형 서점 또는 개별 브랜드의 사이트에서 구매 
   
     ➡️ 한 곳에 다양한 분야의 잡지들을 직접 비교할 수 있는 잡지 전문 온라인 서점
   
   - 종이 잡지 시장의 규모 감소로 인한 해결책 모색 
   
     ➡️ 온라인 서점에서 유저들끼리 잡지 정보를 공유할 수 있는 커뮤니티 서비스를 제공하여 구매 유도
 
 - 개발 환경
 
   - OS : Window 10

   - 개발 언어 : JAVA 11, HTML5, CSS3, JavaScript, jQuery, JSP, SQL

   - 개발 도구 : Eclipse IDE (version 2021-09 (4.21)), Visual Studio Code, SQL Developer

   - Modeling Tool : ERDCloud, StarUML, Oven (카카오 Prototyping tool)

   - DB : Oracle 11g Express Edition

   - WAS : Apache Tomcat 9

   - Version management : Git (GitHub Desktop) 

   - 라이브러리 : cos.jar, gson-2.8.5.jar, json-simple-1.1.1.jar, ojdbc6.jar

   - 프레임 워크 : Bootstrap4

   - API : Summernote, I'mport

### 📚회고

- 좋았던 점

  항상 사용자 측면에서 웹 페이지를 사용해왔는데 내가 사용했던 기능들을 처리하기 위해서는 관리자 입장에서 어떤 방식으로 해당 기능이 구현되고 관리하는지 알 수 있는 기회였다. 프로젝트 이전에는 JAVA, SQL, JDBC, FRONT, SERVER 등 각각의 개념을 익혀왔다면 해당 프로젝트를 통해서 지금까지 배웠던 내용들을 통합해서 다룰 수 있어서 전체적으로 내가 어떠한 부분이 부족하고 어려워하는지 알게 되었다. 
  
  특히 MVC 개념을 이해하는 것에 어려움을 느꼈는데 프로젝트를 하며 계속해서 반복하니 자연스럽게 Controller, Service, Dao, View, DB의 역할과 흐름을 이해하게 되었다. 이론으로 개념을 정립하는 것도 중요하지만 실습만큼 좋은 것은 없다는 걸 몸소 느끼게 된 계기였다.

- 아쉬운 점 
 
  아무래도 처음 하는 프로젝트여서 개인적으로 어려움도 많았고 아쉬움이 남는다. 먼저 DB 설계 당시 테이블 수를 줄이고 하나의 테이블에 관련된 모든 칼럼을 생성하니 기능 구현할 때 테이블을 새로 만든다든지, 칼럼 값을 변경하는 일이 잦아 시간을 많이 뺏겼다. 프로젝트 전에도 작은 홈페이지 형식으로 여러 번 실습을 하였지만 막상 프로젝트가 시작되니 큰 틀로 보면 CRUD 하는 것은 같지만 그 안에 내가 구현하려는 기능을 적용해서 구현하는 것에 어려움을 느꼈다. 
  
  또한 UI 구현 당시 설계도와 수정되는 점이 많아 기능 구현 시간이 줄어들었다. 내가 해당 페이지에서 어떠한 기능을 구현할지에 따라 UI의 모습이 달라야 하는데 기능은 생각하지 않고 평소에 내가 접했던 UI의 모양만 따라 하려고 하니 해당 시행착오가 생긴 것 같다. 간단하게는 체크박스가 아니라 라디오 버튼을 사용해야 된다든지, 전체 조회 페이지에서 버튼으로 처리하는 것이 아니라 상세페이지를 만들어서 다른 기능을 추가해야 된다든지 등 UI와 기능을 함께 생각해서 구현해야 된다는 것을 깨닫게 되었다. 
  
  마지막으로 역할분담 기능별로 나누어 한 사람이 해당 기능의 사용자 측면, 관리자 측면을 함께 담당하면 흐름을 파악하고 구현할 때 좀 더 편안하지 않았을까 생각이 든다. 모두가 처음이라 사용자, 관리자 측으로 역할분담을 하였는데 구현하면서 서로 같은 테이블을 사용하다 보니 소통이 많아야 했는데 소통의 부재가 결국 기능의 부재로까지 이어졌다. 프로젝트를 한다는 것은 개인의 실력을 확인하고 성장하는 것도 맞지만 여러 사람들과 함께 하나의 홈페이지를 만들어가는 작업인 만큼 계속해서 소통하고 함께 고민해야 된다는 걸 느꼈다.
